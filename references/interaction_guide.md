# VOC 引导话术指南（不代答）

本文件供 Agent 在交互引导流程中使用。核心纪律：**你可以归纳、给示例、追问细节，但绝不替用户编造反馈、不替用户写 CTQ、不替用户定优先级取舍。**

## 门1：采集反馈

话术示例：
> "我们先收集【分析对象】的顾客反馈。你可以直接贴一段原始反馈（每行一条），也可以上传 .txt/.docx/.xlsx 我帮你解析。如果一时凑不齐，我们一条条聊也行——比如先从你印象最深的抱怨或好评说起。"

若用户给的反馈很笼统（"用户不太满意"）→ 追问："'不太满意'具体体现在哪？是质量、价格、还是服务？有没有原话或场景？"

❌ 禁止："根据行业经验，智能手表的顾客通常抱怨电池和死机，我帮你列上了。"
✅ 正确："你提到的'续航短'是一条真实反馈。还有其他使用场景的反馈吗？比如充电、屏幕、佩戴？"

## 门2：整理分类

Agent 将反馈按维度（质量/性能/外观/价格/服务/交付…）分组、标情感，然后**请你确认**：
> "我把反馈归了三类——① 性能类（续航短、死机）② 外观类（壳有划痕）③ 服务类（售后慢）。你看归类对吗？有没有跨类的？"

禁止擅自改情感倾向：用户说"价格贵但能接受"是中性期望，不是负面。

## 门3：CTQ 转化

对关键需求，引导转成可测量指标：
> "【电池续航短】这条，如果转成 CTQ，你会怎么量化？比如'单次充电续航 ≥ 2 天'？阈值你来定，我给格式建议。"

❌ 禁止："这条 CTQ 我定为续航≥2天。"
✅ 正确："CTQ 要可测量（带时间/数量/比例）。你希望阈值是多少？我帮你写成标准格式。"

## 门4：优先级

基于 IPA（重要度×满意度）给象限建议，但取舍交用户：
> "按反馈频次和你的判断，'续航'属于【高重要-低满意】应优先改善；'外观颜色'是【低重要-高满意】可维持。但先改哪个最终你定，我给的是建议。"

## 跳过/边界

1. 用户让 Agent 编反馈 → 不可，说明 VOC 价值在真实声音。
2. 用户跳过某条 CTQ → 标注「供参考·待确认」。
3. 战略未提供 → 优先级标注「待提供战略后复核」。
